# Bekannte Gotchas

**SQLite DATE-Felder:** API gibt Datumsfelder als ISO-Timestamp zurück (`"2026-05-30T00:00:00Z"`). Im Frontend immer `.slice(0, 10)` für Vergleiche und `date + 'T12:00:00'`-Konstruktionen.

**Aktive Saison:** Spielplan, Dienst-Erstellung und Dienst-Konten setzen eine aktive Saison voraus (Verwaltung `/admin/saisons`). Ohne aktive Saison schlagen game- und slot-Inserts mit FK-Fehler fehl.

**SSE Live-Updates:** Jede Mutations-Route (`POST`/`PUT`/`DELETE`) muss `h.hub.Broadcast("domain-event")` aufrufen; das Frontend abonniert mit `useLiveUpdates((event) => { if (event === 'domain-event') reload() })`. `Handler`-Structs mit Mutationen brauchen ein `hub *hub.EventHub`-Feld (in `main.go` via `NewHandler(db, hub)` übergeben). Fehlt `Broadcast` (Backend) **oder** `useLiveUpdates` (Frontend), bleibt die Seite nach fremden Änderungen stumm.

**Push Notifications:** Infrastruktur in `internal/notifications/`. VAPID-Keys via `go run ./cmd/teamwerk gen-vapid` in `.env` (`VAPID_PUBLIC_KEY`, `VAPID_PRIVATE_KEY`, `VAPID_EMAIL`). Senden immer als Goroutine (darf den HTTP-Response nicht blockieren):

```go
go notifications.SendToUsers(h.db, h.cfg, []int{userID1, userID2}, "Titel", "Text", "/ziel-url")
```

Frontend-Hook `usePushSubscription` (in `AppShell.tsx`) registriert automatisch beim App-Start. iOS nur als Homescreen-PWA (`display-mode: standalone`). Subscriptions in `push_subscriptions`; ungültige Endpoints (HTTP 410) werden bereinigt. Scheduled Notifications → Job im `internal/scheduler/`, idempotent via `notification_log`.

**App-Icon-Badge:** Chat-Pushes (`chat.Handler.SendMessage`, `SendBroadcast`) berechnen pro Empfänger via `chat.ComputeUnreadForUser` den aktuellen Chat-Unread und senden ihn als absolutes Feld `badge: number` in der Payload — versendet über `push.SendToUserWithBadge` statt `push.SendToUsers`. Der Service Worker (`web/src/sw.ts`) setzt damit `navigator.setAppBadge`/`clearAppBadge`. Im offenen Frontend setzt `AppShell` den Badge zusätzlich live über `useEffect([chatUnread])`. Andere Push-Caller (Games/Trainings/Duties) bleiben bei `push.SendToUsers` ohne `badge`-Feld — der Service Worker rührt den Badge dann nicht an.

**Dienst-Anleitungen (Markdown):** Pro `duty_types` liegt eine optionale Markdown-Anleitung in `instruction_md`; Vorstand pflegt sie auf `/admin/duty-types` (Modal). Der Slot-Link (`<BookOpen>` in `DutySlotList`) erscheint nur, wenn `has_instruction=true` im `GET /api/duty-board`-Response gesetzt ist. Anleitungs-**Bilder** werden per `![Alt](/dokumente/datei/{fileId})` referenziert — Konvention: Ordner **„Anleitungen"** in `/dokumente` mit `folder_permissions.everyone.can_read=1` anlegen, sonst Broken-Image für Nutzer ohne Ordner-Zugriff. Renderer ist `MarkdownRenderer` mit `rehype-sanitize` (kein rohes HTML).

**Auto-Duty-Regen:** Jede Spieländerung (`POST/PUT/DELETE /api/games/{id}`) triggert Regeneration der Dienst-Slots für Event-Datum ± 1 Tag (beachtet `same_day_behavior`/`adjacent_day_behavior` der Duty-Types). Slots mit `is_custom=1` (manuell angelegt/editiert) werden geschont. Response enthält `regen_summary`. Vor Deploy manuell-editierte Bestandsslots mit `UPDATE duty_slots SET is_custom=1 WHERE id IN (...)` schützen. **`games.template_id IS NULL` heißt explizit „keine Auto-Dienste"** für *alle* `event_type` (heim/auswärts/generisch) — es gibt keinen ID-basierten Fallback mehr. Beim `PUT` ist `template_id` Tri-State: Feld fehlt = unverändert, `null` = NULL setzen, Zahl = setzen. Generische Events nutzen Templates mit `template_type='generisch'`; die Spieldauer kommt dann aus `game_templates.duration_minutes` statt aus der Altersklasse. Für generische Events koexistieren `is_custom=1`-Slots (aus `slots[]` im Request) und Template-`is_custom=0`-Slots; Doppelungen werden im `regen_summary.conflicts` gemeldet.

**Video-Streaming (HLS + AirPlay + Chromecast):** Der Transcode erzeugt **nur noch 720p** (keine 360p-Fallback-Rendition mehr — bewusster Speicher-Trade-off, kein ABR bei schwacher Verbindung). ffmpeg läuft mit `-pix_fmt yuv420p` (sonst produziert libx264 bei 10-bit-iPhone-Quellen yuv420p10, das tvOS nicht dekodiert). Nach dem 720p-Lauf probet `probeSegmentCodecs` (`internal/videos/codecs.go`) das erste Segment und leitet den `CODECS`-String (`avc1.PPCCLL,mp4a.40.X`) ab, der in `videos.codecs` gespeichert und in `master.m3u8` als `#EXT-X-STREAM-INF:…,CODECS="…"` gehängt wird — **Pflicht für AirPlay/tvOS**, sonst kommt Ton ohne Bild. Zusätzlich `#EXT-X-INDEPENDENT-SEGMENTS`. Bestandsvideos wurden per einmaligem, idempotentem Backfill (`internal/videos/backfill.go`, gestartet als Goroutine in `serve()`) migriert: Query `status='ready' AND codecs IS NULL`, ffprobe auf `720p/seg_001.ts` (Fallback `360p/seg_001.ts`), master neu schreiben, altes 360p-Verzeichnis löschen. **Stream-Token-TTL** ist an die Video-Dauer gebunden: `clamp(duration_sec + 30min, 1h, 4h)` — Kurzclips 1 h, Vollspiele bis 4 h. `Play` übergibt `int(video.DurationSec)` an `Sign`. Videos ohne `duration_sec` behalten die Legacy-1h. **CORS**: `ServeMaster`/`ServeRenditionFile` setzen `Access-Control-Allow-Origin: *` und `Access-Control-Allow-Methods: GET`; OPTIONS-Preflight (`HLSPreflight`) läuft ohne Auth durch die `StreamTokenMiddleware`. Auth bleibt der `?st=`-Token — keine Cookies. **Chromecast-Sender im Frontend** (`web/src/lib/cast.ts` + `components/CastButton.tsx`): Cast-SDK von gstatic wird **erst nach Klick** geladen (DSGVO-neutral im Default). Kein `integrity=`-SRI — Google versioniert das Skript serverseitig ohne stabile Hashes und pusht Firmware-Kompat-Updates; ein fixer Hash würde bei jedem Google-Push brechen. Mitigation: `crossorigin="anonymous"` + explizite User-Aktion. `<video>` trägt zusätzlich `x-webkit-airplay="allow"` als belt-and-suspenders.

**Video-Upload Progress-Throttle:** `onProgress` in tus-in-Browser MUSS throttlen — `XHR.upload.progress` feuert 30–100 Hz; jeder Event trigert sonst zwei React-`setState`, der Reconciler teilt sich den Main-Thread mit dem XHR-Reader → **5–10× Throughput-Verlust** bei großen Uploads (bei 9 GB: Stunden statt Minuten). `VideoUploadPage.tsx` löst das über eine Modul-Level-Factory `createThrottledProgress({ setProgress, setRemaining })`: max. 1 State-Update/s, `setProgress` zusätzlich mit Prozent-Guard (unveränderter gerundeter Wert → kein Update), Restzeit-Schätzung aus Sliding-Window der letzten ~10 s (reagiert schnell nach Bandbreiten-Aussetzern, ist nicht der Gesamt-Durchschnitt seit Start). Pro Upload eine frische Factory-Instanz aufrufen — Closure-State ist dann automatisch sauber (kein manuelles Reset). Tests in `VideoUploadPage.test.tsx`; die Factory ist bewusst exportiert für Unit-Testbarkeit ohne Component-Render.

**Video-Upload Auth-Refresh:** Der tus-Upload (`web/src/pages/VideoUploadPage.tsx`) läuft über `fetch`/`XHR`, **nicht** über die axios-Instanz — der 401-Auto-Refresh aus `lib/api.ts` greift dort also nicht. Statt eines statischen `headers`-Objekts (fror den Access-Token beim Upload-Start ein → 401 nach 15 min) setzen zwei tus-Hooks (`buildAuthHooks()`) den Bearer-Token: `onBeforeRequest` liest ihn pro Request frisch aus dem Store (und **wartet** einen laufenden Refresh ab, damit der Retry den neuen Token trägt), `onShouldRetry` stößt bei einem Chunk-401 `refreshAccessToken()` an (denselben Single-Flight-Guard wie axios) und retryt. **Wichtig:** `onShouldRetry` muss **synchron** ein `boolean` liefern — tus 4.x wertet den Rückgabewert direkt in `if (shouldRetry(...))` aus; eine `async`-Variante lieferte ein (immer truthy) Promise und hebelte den Abbruch-Pfad aus. Scheitert der Refresh selbst mit 401 (Refresh-Token abgelaufen): `setAccessToken(null)` + Redirect auf `/login`.

**SEPA-XML-Schema (pain.008.001.08):** Verpflichtend ist das aktuelle **DK-TVS** — konkret welche Version, Herkunft und Upgrade-Prozess siehe `web/src/lib/__schemas__/README.md` (Single Source of Truth). Jede Änderung an `web/src/lib/sepaXml.ts` wird von `web/src/lib/sepaXml.xsd.test.ts` mechanisch gegen dieses Schema geprüft (Vitest shell-outet `xmllint`). Lokal ohne `xmllint` (libxml2) skippt der Test mit sichtbarem Hinweis; im CI-Gate (`.github/workflows/ci.yml`) ist `libxml2-utils` verpflichtend installiert und die Env-Var `TEAMWERK_REQUIRE_XSD=1` sorgt dafür, dass fehlendes xmllint dort hart fällt statt still zu skippen. **Bank-relevante Härtungen im Generator** (weil DK-TVS strenger als ISO-XSD): `<Nm>` auf 70 Zeichen begrenzt (Debtor/Creditor/InitgPty, Truncation-Events werden als `warnings` an den Aufrufer zurückgegeben — Kassierer bestätigt vor Download), `<Ustrd>` auf 140, `<PstlAdr>` all-or-nothing (weglassen wenn `city` fehlt, sonst mit `<TwnNm>` + `<Ctry>`), `<DtOfSgntr>` fällt bei fehlendem Mandatsdatum auf `DEFAULT_MANDAT_DATUM` (Kommentar in `sepaXml.ts` dokumentiert Trade-off), `betragCent <= 0` wirft (kein Silent-Math-Abs).

**SEPA-Beitragslauf:** `/admin/beitragslauf` (`vorstand`, `kassierer`, `admin`). Bewusst einfach: **keine monatsgenaue Pro-rata-Berechnung**, Fälligkeit **immer 01.07.** der Saison, alle Lastschriften **RCUR** (keine FRST), Spieler gelten als Kinder. **Exakte Halbierung (`betrag_cent / 2`)** statt vollem Beitrag, wenn eine von drei Bedingungen greift (Ermäßigungen stapeln **nicht**, Priorität in dieser Reihenfolge): (1) Saison ist `seasons.is_inaugural` (erstes Abrechnungsjahr des Vereins, einmalig → **alle** halb); (2) `members.join_date` liegt im Saisonfenster `[start_date, end_date]` (unterjähriger Eintritt); (3) `members.status='ausgetreten'` **und** `members.exit_date` im Saisonfenster (unterjähriger Austritt). Unterjährige Austritte werden — abweichend von honorar/anwaerter — **wieder in den Lauf einbezogen** (Kategorie aus `home_club_id` wie aktiv, halbiert); früher Ausgetretene bleiben unsichtbar. `members.join_date` ist Pflichtfeld (Anlage/Bearbeitung, HTTP 400 sonst), `exit_date` Pflicht bei Status `ausgetreten`; Bestandsmitglieder bekamen per Migration `014` ein implizites `join_date` **vor** dem ersten Saisonstart (→ voller Beitrag ab Jahr 2). Im SEPA-XML-Verwendungszweck steht nur das **Beitragsjahr** (z. B. `26`), nicht die Saison-Spanne (`web/src/lib/sepaXml.ts`, `abrechnungsjahr()`). Vor dem ersten Lauf müssen die SEPA-Stammdaten (`glaeubiger_id`, `iban`, `bic`, `kontoinhaber`) unter Einstellungen → Verein gepflegt sein, sonst liefert `POST /api/fee-run/export-data` HTTP 400. **Zero-Knowledge (Modell B):** der Kassierer muss den **Bankdaten-Tresor entsperren** (Menü „Tresor"); `export-data` liefert nur Ciphertext + Wraps, die `pain.008`-Datei wird **im Browser** erzeugt (`web/src/lib/sepaXml.ts`, IBAN-Prüfung `lib/sepa.ts`) — der Server sieht keine Klartext-IBAN. Beitragsmatrix unter Einstellungen → Beiträge (3 Kategorien, Cent, Historie via `valid_from`). „Lauf bestätigen" schreibt das append-only Saison-Protokoll (nur member_id/Betrag/Erfolg, keine IBANs). Kassierer korrigiert Bankdaten via `PUT /api/members/{id}/bank-details` (Envelope, clientseitig verschlüsselt).

**Chat-Read-Receipts (Absender-Sicht):** WhatsApp-artige Häkchen pro eigener Nachricht. `message_reads.read_at` existiert seit Migration 001 (Default), Migration `035` ergänzt nur `idx_message_reads_message_id`. `ListMessages` liefert pro Nachricht `readCount` (Leser außer Sender), `readTotal` (aktive Mitglieder außer Sender) + `read` (`readCount>0` → Direct kollabiert auf gesendet/gelesen). Live: `MarkRead` (`POST /chat/conversations/{id}/read`) erfasst die neu zu markierenden Nachrichten **vor** dem `INSERT OR IGNORE` (danach ist neu/Bestand nicht mehr unterscheidbar), gruppiert nach Absender und feuert pro Absender **genau ein** coalesced Hub-Event `chat:read-receipt:<convId>:<readerUserId>:<upToMessageId>` (kein N-Event-Sturm bei Bulk-Read). Chat-Events sind **plain colon-strings** (kein JSON-Payload) über den eigenen SSE-Kanal `/api/chat/events`; das Frontend parst mit `split(":")` und markiert eigene Nachrichten `id ≤ upTo` als gelesen. Detail-Liste `GET /api/chat/messages/{id}/reads` ist **Absender-only** (403 sonst, 404 bei fehlender/gelöschter Nachricht) — Reads Dritter bleiben für andere Empfänger unsichtbar. Broadcasts sind bewusst außen vor (eigener Change).
