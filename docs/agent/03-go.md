# Go-Konventionen

**Handler-Pattern** — ein Package pro Domäne unter `internal/`:

```go
type Handler struct{ db *sql.DB; hub *hub.EventHub }
func NewHandler(db *sql.DB, hub *hub.EventHub) *Handler { return &Handler{db, hub} }
func (h *Handler) MethodName(w http.ResponseWriter, r *http.Request) { … }
```

**Router (Chi v5):** `r.Get("/api/members/{id}", membH.Get)`, auslesen mit `r.PathValue("id")`.

**Auth/JWT:** `claims := auth.ClaimsFromCtx(r.Context())`. Access-Token 15 min (HS256, im Frontend im Memory), Refresh-Token 7 Tage (opaque, SHA256-Hash in DB, HttpOnly-Cookie). Gating via `auth.RequireRole(...)` und `auth.RequireClubFunction(...)`.

**DB-Zugriff:** Kein ORM. Bei DB-Open automatisch `PRAGMA journal_mode=WAL; foreign_keys=ON;`. Nullable Felder über `sql.NullInt64` & Co. scannen, dann `.Valid` prüfen.

**Bank-/SEPA-PII (Zero-Knowledge, clientseitig verschlüsselt — Modell B):** Mitglieds-Bankdaten (`member_sensitive.ciphertext` + `dek_enc_vorstand`), Vereins-SEPA-Stammdaten (`clubs.sepa_ciphertext`/`sepa_dek_enc`), `member_change_drafts` mit `field_name='bankdaten'` (Envelope im `new_value`) und SEPA-Mandat-PDFs (Datei-Blob + `members.sepa_mandat_dek_enc`) werden **im Browser** verschlüsselt; der **Server sieht nie Klartext und besitzt keinen Entschlüsselungsschlüssel**.
- **Krypto:** asymmetrisches Vereins-Keypair (RSA-OAEP). Der öffentliche Schlüssel (`clubs.group_public_key`, via `GET /api/encryption-pubkey`, nicht geheim) wrappt pro Datensatz einen zufälligen AES-GCM-DEK → **jeder** kann schreiben (Mitglied/Eltern/Beitritt). Der private Schlüssel liegt `AES-GCM(PKCS8, PBKDF2(passphrase))`-verschlüsselt (`clubs.group_private_key_enc`) und entschlüsselt nur mit der geteilten Tresor-Passphrase (`vorstand`/`kassierer`) → **nur die Finance-Gruppe liest**. Frontend: `web/src/lib/crypto.ts` (Primitive) + `bankCrypto.ts` (Envelope) + `VaultContext` (entsperrter Privatschlüssel, sessionStorage, 30-min-Timeout).
- **Server-Regeln:** Schreib-Routen nehmen `{*_ciphertext, *_dek_enc}` entgegen und **lehnen Klartext-Bankfelder mit 400 ab**; Lese-Routen liefern den Envelope nur an admin/vorstand/kassierer (kein Server-Decrypt; kein `policy.CanDecryptBankData` mehr). Eigentümer/Eltern lesen Bankdaten **nicht** zurück (G2). IBAN-Validierung clientseitig (`web/src/lib/sepa.ts`); SEPA-`pain.008` baut der Browser des Kassierers (`web/src/lib/sepaXml.ts`, `POST /api/fee-run/export-data` liefert nur Ciphertext).
- **Modellgrenze:** schützt gegen passive Kompromittierung (Backup/Snapshot/Hoster/Admin), **nicht** gegen aktiv bösartigen Server (liefert das JS aus) oder Endgerät. Kein Recovery — Passphrase-Verlust = Datenverlust. `internal/crypto` (`Decrypt`/`DecryptBytes`, `FIELD_ENCRYPTION_KEY`) existiert nur noch für die **einmalige Migrations-Brücke** des Altbestands (siehe Deployment).

**E-Mail:** `h.mailer.Send(to, subject, body)` (net/smtp, Config aus `.env`).

## Rollen und Vereinsfunktionen (zwei orthogonale Dimensionen)

**System-Rolle** (`users.role`, JWT-Claim `role`, via `auth.RequireRole(...)`):

| Rolle | Bedeutung |
|---|---|
| `admin` | Vollzugriff; umgeht alle `RequireClubFunction`-Checks |
| `standard` | Default; Zugriff aus Vereinsfunktionen + Ownership |

**Vereinsfunktion** (`member_club_functions.function`, 0–n pro Member, JWT-Claim `club_functions: string[]`, via `auth.RequireClubFunction(...)` / `claims.HasFunction(...)`): `spieler`, `trainer`, `sportliche_leitung`, `vorstand`, `vorstand_beisitzer`, `kassierer`.

- **Eltern** sind keine Vereinsfunktion → `claims.IsParent` (aus `family_links`), nie `HasFunction("elternteil")`.
- `sportvorstand` existiert **nicht** (= `vorstand` + `sportliche_leitung`).
